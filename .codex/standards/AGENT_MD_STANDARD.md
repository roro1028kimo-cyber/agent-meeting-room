# AGENT_MD_STANDARD｜Codex 全域 agent.md 寫作格式

> 這份文件定義 Codex 撰寫 `agent.md` 的標準格式。
> 目標不是寫得漂亮而已，而是讓 `agent.md` 具備「可掃描、可對齊、可直接開發」三個條件。

---

## 一、標準定位

| 欄位 | 規格 |
|------|------|
| **文件名稱** | `agent.md` |
| **文件用途** | 定義產品定位、邊界、模組、資料結構、流程與 MVP 驗收 |
| **適用情境** | 新專案立項、舊專案重構、功能方向重整、多人協作對齊 |
| **寫作語言** | 預設繁體中文 |
| **文件風格** | 規格書型，不是腦力激盪筆記，也不是散文說明 |
| **參考風格** | 採用表格化、邏輯化、可掃描的規格書寫法 |

---

## 二、寫作原則

### 必須遵守

1. **開頭先定義產品身分**
   第一屏必須先回答「這是什麼」、「不是什麼」、「定位在哪裡」。

2. **先講原則，再講功能**
   不可以一開始就堆功能清單，要先把設計原則寫清楚。

3. **一定要有邊界**
   `In scope` 和 `Out of scope` 都要清楚，避免 scope creep。

4. **資訊要可掃描**
   能用表格就不要只用長段落。

5. **技術決策要有理由**
   不只寫「用什麼」，還要寫「為什麼這樣選」。

6. **要可直接開工**
   文件必須包含模組結構、資料模型、流程或 API 等足以啟動開發的資訊。

7. **MVP 要能驗收**
   一定要有 checklist，不可只寫抽象願景。

### 明確禁止

- 只寫願景，不寫結構
- 只寫功能，不寫邊界
- 重複段落過多
- 一個章節混太多主題
- 寫成聊天紀錄式筆記

---

## 三、標準章節順序

每一份 `agent.md` 原則上都應遵循以下順序：

1. 文件標題與一句話描述
2. 角色定義
3. 產品原則
4. 問題定義
5. 與既有系統或外部系統的關係
6. 開發環境
7. 技術選型
8. 模組結構
9. 核心資料結構
10. 核心流程規格
11. UI 規格
12. 設定規格
13. 角色系統或權限系統
14. 記憶 / 儲存 / 匯出規格
15. API 合約
16. MVP 驗收清單
17. Sprint 或 Phase 規劃
18. 未來方向
19. 一句話定位與差異化
20. 總結

---

## 四、標準語氣

`agent.md` 應使用以下語氣：

- 明確
- 冷靜
- 不繞圈
- 直接定義
- 可被工程與產品共同閱讀

不應使用以下語氣：

- 太像行銷稿
- 太像聊天回覆
- 太像開發日誌
- 太像零碎備忘錄

---

## 五、標準排版規則

### 標題規則

- 使用 `# / ## / ###`
- 大章節固定用 `##`
- 子章節固定用 `###`

### 表格規則

以下資訊優先用表格表示：

- 產品身份卡
- 技術棧
- 狀態列舉
- 角色清單
- API 清單
- Sprint 清單
- 比較表

### 條列規則

以下資訊優先用編號或項目清單表示：

- 流程
- 原則
- 驗收條件
- 不做清單

### 程式碼區塊規則

若有以下內容，可放入程式碼區塊：

- 模組樹
- API 範例
- 資料結構範例
- 設定範例

---

## 六、每章節應回答的問題

### 1. 角色定義

要回答：

- 產品叫什麼
- 產品是什麼
- 產品不是什麼
- 它服務誰
- 與外部系統的關係

### 2. 產品原則

要回答：

- 這個產品優先保護什麼
- 它的核心設計選擇是什麼
- 哪些事情這一版刻意不做

### 3. 問題定義

要回答：

- 市面上現有方案哪裡不夠
- 這個產品解決什麼問題
- 為什麼現在要做

### 4. 技術選型

要回答：

- 前端為什麼這樣選
- 後端為什麼這樣選
- 資料庫為什麼這樣選
- 哪些選型是 MVP 策略，不是永遠策略

### 5. 模組結構

要回答：

- 專案主要檔案怎麼分
- 每個模組大概負責什麼
- 哪些是主流程模組

### 6. 資料結構

要回答：

- 核心實體有哪些
- 每個實體的責任是什麼
- 狀態或型別如何枚舉

### 7. UI 規格

要回答：

- 使用者進來第一眼看到什麼
- 哪個區域是視覺中心
- 哪些互動必須被清楚區分

### 8. API 合約

要回答：

- 前端需要哪些 API
- 每個 API 對應什麼使用者操作
- 路徑命名是否直觀

### 9. MVP 驗收

要回答：

- 什麼叫做第一版完成
- 哪些功能能跑才算完成
- 哪些測試一定要過

## 八、Codex 執行規則

之後 Codex 在撰寫任何 `agent.md` 時，應預設遵守以下規則：

1. 優先使用繁體中文。
2. 優先寫成規格書，不寫成說明信。
3. 優先提供表格、清單、模組樹與驗收清單。
4. 先定義產品，再定義功能。
5. 若有外部系統，必須明確說明「整合」還是「重做」。
6. 若文件讀起來還像備忘錄，就代表還沒寫到位。

---

## 九、判斷文件是否合格

一份合格的 `agent.md`，應讓人快速回答以下問題：

- 這是什麼產品
- 它解決什麼問題
- 它不做什麼
- 它用什麼技術
- 它的模組怎麼分
- 它的資料怎麼存
- 它的流程怎麼跑
- 什麼算第一版完成

如果讀完整份文件後，還不能回答以上問題，代表文件不合格。
