# Agent Meeting Room

## 文件目的

本文件定義本專案的新版本方向：一個**輕量化、多 agent、單機優先**的 AI 會議室。

這個版本不再把重點放在完整專案管理流程，也不追求複雜的 PM 狀態機。核心目標只有一件事：

讓多個 AI agent 可以在同一個會議室介面中，針對一個主題穩定地開會、輪流發言、收斂結論，並在會後把結果存成可讀與可重用的長期記憶。

## 一句話定義

這是一個本地優先的多 agent 會議室，使用者只需要一個介面，就能建立議題、選擇與設定 agent、啟動討論、查看發言、匯出結果，並保留會後記憶。

## 版本定位

本版產品定位如下：

- 單機優先
- 功能輕量
- 一個介面，多個 agent
- 設定可編輯
- 角色可自訂
- 會議過程用暫存記憶
- 會後可匯出並進入長期記憶

本版**不是**：

- 完整專案管理系統
- 複雜工作流平台
- 多租戶協作後台
- 重度權限管理系統
- OpenClaw 的重製版

## 核心目標

本版至少要達成以下目標：

1. 使用者可以建立一場會議
2. 使用者可以選擇多個 agent 參與討論
3. 每個 agent 可以依自己的定位輪流發言
4. 主持人或總結角色可以整理結論
5. 使用者可在會議中插話補充
6. 會議結果可匯出成文字與 Python 格式
7. 匯出後的結果可存檔並進入長期記憶
8. 整套系統可以在單機穩定運行

## 非目標

為了維持輕量與穩定，本版不做以下內容：

- 複雜 queue 優先級機制
- 專案里程碑管理
- blocker、風險、驗收等完整 PM 欄位模型
- 多使用者帳號系統
- 即時多人協作
- 附件解析
- 自動週報
- 高度自動化工具鏈
- 重新打造 OpenClaw Gateway 或 Skills 系統

## 與 OpenClaw 的關係

本專案未來可接入 OpenClaw，但不重做以下能力：

- OpenClaw Gateway
- OpenClaw 多 agent 路由
- OpenClaw Skills 載入與註冊系統
- OpenClaw 工作區與 agentDir 管理

本專案把 OpenClaw 視為：

- 可整合的外部 agent 來源
- 可被使用的技能與角色生態
- 未來可接入的多 agent 執行環境

因此本版策略是：

- 先把會議室本身做輕、做穩
- 設計好「外部 agent / OpenClaw agent」席位概念
- 保留設定欄位與資料結構
- 不在 MVP 中強行假設 OpenClaw API 協定

## 核心使用情境

### 1. 單機快速開會

使用者輸入一個主題與背景，選擇 3 到 6 個 agent，直接開始討論。

### 2. 角色對比討論

同一個議題由不同定位的 agent 發言，例如支持者、反方、規劃者、風險官、記錄員。

### 3. 會後存檔

會議結束後匯出為文字或 Python 格式，存成檔案並加入長期記憶。

### 4. 未來接入 OpenClaw

未來可把某些席位改成 OpenClaw 外部 agent，使會議室成為多 agent 協作 UI。

## 核心原則

### 1. 先穩定，再複雜

本版只保留會議室最必要的能力，不預先做過重流程。

### 2. 一個介面，多個 agent

使用者不需要切換多個頁面或多個聊天室，所有角色都在同一個會議室中發言。

### 3. 設定可見、可改

API、模型、角色定位、角色提示詞都要在設定功能中可見且可編輯。

### 4. 角色是可配置資產

角色不是寫死在程式裡的黑盒，而是可調整名稱、定位與 prompt 的設定項。

### 5. 暫存記憶與長期記憶分開

開會過程中的記憶屬於暫存記憶；只有在會後匯出並存檔後，才進入長期記憶。

### 6. local-first

本版預設以單機運行為主，降低部署與信任成本。

## UI 結構

本版介面採單一會議室畫面，分成以下區塊：

### 1. 主討論區

主視覺是討論內容流，負責顯示：

- 會議主題
- 使用者輸入
- agent 逐位發言
- 正在輸入或思考中的狀態
- 主持人或總結內容

### 2. 右側發話人區

右側顯示所有與會 agent：

- 名稱
- 定位
- 是否啟用
- 發言中狀態
- 綠燈提示
- 使用來源（內建 / 自訂 / OpenClaw 預留）

### 3. 下方輸入區

下方固定輸入區包含：

- 正式會議輸入
- 使用者插話
- 啟動下一輪
- 結束會議
- 匯出

### 4. 設定功能

設定可以獨立為抽屜或面板，不應干擾主會議畫面。

## 設定功能

設定功能至少要包含：

### 1. API 設定

- API 模式
- API Key
- Base URL
- 模型名稱
- 溫度或其他簡化參數

本版優先支援：

- mock 模式
- OpenAI-compatible 模式

### 2. 角色設定

每個角色應可設定：

- 名稱
- 顏色或識別樣式
- 角色定位
- 系統提示詞
- 是否啟用
- 模型覆寫（可選）
- 來源類型（內建 / 自訂 / OpenClaw 預留）

### 3. OpenClaw 預留設定

本版只保留欄位，不要求真正打通：

- OpenClaw 啟用開關
- Gateway URL
- Agent ID 對應資訊
- 備註

## 內建角色建議

本版內建一批可直接使用的角色模板，使用者可在設定中編輯：

### 必備角色

- 主持人
- 規劃師
- 反方辯手
- 風險官
- 執行官
- 記錄員

### 建議內建角色

- 研究員
- 技術顧問
- 產品顧問
- 商業顧問
- 品質官
- 溝通編修
- 創意探索者

### 各角色共通要求

- 發言要短
- 先說重點
- 不講空話
- 不假裝知道不存在的資訊
- 以會議推進為優先

## 會議流程

本版流程刻意簡化如下：

1. 使用者建立會議
2. 使用者輸入主題與背景
3. 使用者選擇參與的 agent
4. 使用者啟動一輪會議
5. agent 逐位發言
6. 主持人或系統輸出本輪摘要
7. 使用者可補充再開下一輪
8. 會議結束後匯出與存檔

不做：

- intake / confirming / reframing 的複雜狀態流
- 高中低優先插話流程
- 複雜回合控制

## 暫存記憶與長期記憶

### 暫存記憶

開會中的記憶屬於暫存記憶，包含：

- 會議主題
- 背景說明
- 本輪與前幾輪發言
- 使用者最近補充

暫存記憶只服務當前會議，不應自動視為永久知識。

### 長期記憶

只有當使用者執行以下流程後，內容才進入長期記憶：

1. 匯出會議結果
2. 存成檔案
3. 建立長期記憶紀錄

長期記憶至少應保存：

- 會議標題
- 匯出時間
- 匯出格式
- 檔案路徑
- 內容摘要

## 匯出規格

本版至少支援兩種匯出格式：

- 文字格式
- Python 格式

### 文字格式用途

- 會議紀錄
- 摘要留存
- 交給人閱讀

### Python 格式用途

- 結構化保存
- 方便之後程式再利用
- 可作為記憶載入、測試資料或二次處理來源

## 資料模型建議

本版資料結構建議簡化為：

- 應用設定
- 角色設定
- 會議
- 會議訊息
- 長期記憶

本版不再要求：

- 複雜快照
- PM 狀態歷程
- queue item
- action item
- risk item

## MVP 範圍

本版最小可用產品只做：

- 單機可啟動
- 一個會議室畫面
- 多個 agent 參與會議
- 可編輯 API 與角色設定
- 可逐輪會議
- 可顯示暫存記憶內容
- 可匯出文字 / Python
- 可寫入長期記憶

## 成功標準

如果本版成功，至少應滿足：

- 使用者可以在 3 分鐘內開始一場會議
- 多個 agent 能穩定發言
- 設定功能可改 API 與角色提示詞
- 會議結果可匯出與存檔
- 本地端不依賴雲端資料庫也能運行
- 產品定位清楚，不再像重 PM 平台

## 最後原則

這個產品最重要的不是功能多，而是：

- 穩定
- 輕量
- 好理解
- 好擴充

若未來要擴充，也應優先遵守以下順序：

1. 先維持會議室本身穩定
2. 再增加角色與 prompt 管理
3. 再接更多模型
4. 最後才考慮更重的技能與外部整合
